## Bandit 18 to 19

login : `ssh bandit18@bandit.labs.overthewire.org -p 2220`  
password : `pGozC8kOHLkBMOaL0ICPvLV1IjQ5F1VA`  

### Task 
The password for the next level is stored in a file `readme` in the home directory. Unfortunately, someone has modified `.bashrc` to log you out immediately whenever you try to log in with SSH.

### Quick theory 

When you log into a server via SSH normally, it spawns an interactive shell and runs startup scripts like `.bashrc` or `.bash_profile`. If an administrator (or a challenge creator) puts an `exit` or `logout` command inside that `.bashrc` file, your connection will drop the moment you authenticate.

However, you can bypass this by passing a command directly at the end of your SSH connection string. 

* `ssh user@host [command]`

When you append a command, SSH executes that specific command on the remote machine in a non-interactive shell (skipping the problematic `.bashrc` script) and then returns the output to your local terminal before cleanly closing the connection.

Tools you’ll use:  
`ssh` → to connect and execute remote commands directly.

### Solution

First, if we try to log in normally, the connection instantly closes:
```bash
┌──(sssubhammm ⌘ macbook-air)-[~]
└─$ ssh bandit18@bandit.labs.overthewire.org -p 2220
[...]
Byebye !
Connection to bandit.labs.overthewire.org closed.
```

To bypass this, we send the `ls` command directly through SSH to see what is in the home directory:
```bash
┌──(sssubhammm ⌘ macbook-air)-[~]
└─$ ssh bandit18@bandit.labs.overthewire.org -p 2220 ls
[...]
readme
```

Now that we know the file is called `readme`, we can send the `cat` command through SSH to read its contents:
```bash
┌──(sssubhammm ⌘ macbook-air)-[~]
└─$ ssh bandit18@bandit.labs.overthewire.org -p 2220 cat readme
[...]
cGWpMaKXVwDUNgPAVJbWYuGHVn9zl3j8
```

The output gives us the password for Bandit 19.
